## Context

当前平台已有用户答题的完整数据链路：t_user（用户统计字段）、t_quiz_session（会话级 score/duration/difficulty）、t_quiz_answer（逐题 is_correct/answer_time）。但缺少积分体系和竞争激励机制。项目使用 Spring Boot + MyBatis Plus + MySQL + Redis 技术栈，已接入 Sa-Token 做认证。

## Goals / Non-Goals

**Goals:**
- 实现基于题目难度的积分计算体系
- 实现 8 级段位体系，段位只升不降
- 实现周联赛分组、排名、晋级/降级机制
- 提供联赛相关的 API 接口供前端展示

**Non-Goals:**
- 不实现好友系统或社交功能
- 不实现联赛机器人/AI 填充（用户不足时直接不开组）
- 不实现联赛奖励/徽章系统
- 不实现前端页面（本期仅后端 API）

## Decisions

### D1: 积分存储方案 — MySQL 双写 + Redis 缓存

**选择**: 每次答题结束时，同时写入 t_user.total_points（累计）和 t_weekly_score（周明细）。周排行榜使用 Redis Sorted Set 缓存。

**备选方案**:
- 纯 MySQL 聚合查询排名 → 读多时性能差
- 纯 Redis 存储 → 重启丢失，需持久化

**理由**: MySQL 保证数据持久性，Redis Sorted Set（ZADD/ZRANGE）天然支持排名查询，O(logN) 复杂度。周积分每周重置时清理 Redis key 即可。

### D2: 周一分组使用定时任务

**选择**: 使用 Spring @Scheduled 定时任务，周一 00:00 执行分组逻辑。

**备选方案**: 用户首次答题时懒加载分组 → 实现复杂，且可能出现"先答题后分组"的时序问题。

**理由**: 定时任务逻辑清晰，批量处理效率高，避免并发问题。

### D3: 段位门槛存储在数据库而非配置文件

**选择**: 新建 t_league_tier 表存储段位配置（名称、图标、门槛积分）。

**理由**: 便于后续运营调整门槛数值，无需重新部署。

### D4: 周积分计算公式与累计积分使用同一套规则

**选择**: 单题积分规则统一，双写到两个维度。

```
easy:   答对 +10, 答错 +2
medium: 答对 +20, 答错 +5
hard:   答对 +35, 答错 +8
balanced: 按题实际 difficulty 字段计算
```

**理由**: 规则统一降低理解和维护成本，用户只需理解一套积分逻辑。

### D5: 分组按历史周积分分层

**选择**: 同段位用户按近 4 周平均周积分排序后分层分组，而非纯随机。

**理由**: 避免强者扎堆导致优秀用户被降级，每组内实力接近更公平。

### D6: 人数不足的分档处理

```
30+  → 正常分组，余数不开组
15~29 → 正常开组，比例缩晋级/降级名额
10~14 → 只设晋级 Top 3，不设降级
<10  → 不开组
```

## Risks / Trade-offs

**[R1] 用户量小时联赛体验差** → 通过"人数不足不开组"兜底，避免空组尴尬。未来可考虑 AI 机器人填充。

**[R2] 周日结算定时任务失败** → 增加补偿机制：若周一发现上周未结算，手动触发结算。结算操作设计为幂等可重入。

**[R3] 积分计算与答题提交耦合** → 积分计算放在 finishSession 流程中，若积分写入失败不影响答题完成。使用异步事件或 try-catch 隔离。

**[R4] Redis Sorted Set 与 MySQL 数据不一致** → 以 MySQL 为权威数据源，Redis 仅做查询加速。每周结算时从 MySQL 重新计算排名写入。
