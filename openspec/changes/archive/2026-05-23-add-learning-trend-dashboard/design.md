## Context

当前系统已有学习统计功能，包括：
- 用户连续学习天数（streakDays）
- 答题连对数（streak）
- 总题数、正确率等基础统计

但这些数据只展示当前状态，无法展示历史趋势。用户需要一个看板来可视化自己的学习进步轨迹。

## Goals / Non-Goals

**Goals:**
- 提供按日/周/月维度的学习数据统计
- 展示正确率趋势图表
- 展示学习题量趋势图表
- 展示学习时长统计
- 支持时间范围选择（最近7天、30天、90天）

**Non-Goals:**
- 不支持导出数据（后续功能）
- 不支持多用户对比（仅个人数据）
- 不支持实时更新（按需刷新）

## Decisions

### 1. 图表库选择：ECharts

**选择**: 使用 ECharts 作为前端图表库

**理由**:
- 功能丰富，支持多种图表类型
- 中文文档完善，社区活跃
- 支持响应式和交互功能
- Apache ECharts 是开源免费的

**备选方案**:
- Chart.js：更轻量，但功能相对简单
- D3.js：功能强大但学习曲线陡峭

### 2. 数据统计方式：后端聚合

**选择**: 在后端进行数据聚合统计

**理由**:
- 减少前端数据传输量
- 数据库查询效率更高
- 便于缓存和优化

**备选方案**:
- 前端聚合：传输原始数据量大，前端计算压力大

### 3. API 设计：RESTful

**选择**: 新增 `/api/v1/trend/stats` 端点

**参数**:
- `period`: day/week/month
- `days`: 统计天数（7/30/90）

**返回**: 按时间维度聚合的统计数据数组

### 4. 数据库查询：使用原生 SQL

**选择**: 使用 JPA 的原生 SQL 进行聚合查询

**理由**:
- 复杂的日期分组和聚合用原生 SQL 更灵活
- 性能可控

**备选方案**:
- JPA Criteria API：复杂聚合查询代码冗长
- QueryDSL：需要额外依赖

## Risks / Trade-offs

| 风险 | 缓解措施 |
|------|----------|
| 大量历史数据查询性能问题 | 添加数据库索引；限制最大查询范围（90天） |
| ECharts 包体积较大 | 按需引入，使用 tree-shaking |
| 时区问题影响日期统计 | 统一使用用户本地时间，后端存储 UTC |

## Migration Plan

1. 后端：新增 API 和查询方法
2. 前端：新增页面和图表组件
3. 数据库：添加必要索引（如有性能问题）
4. 无需数据迁移，纯新增功能
